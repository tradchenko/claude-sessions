# Plan 02-02: Система структурированных ошибок и exit codes

## Контекст

Текущие ошибки — `console.error(❌ ${t(key)})` без структуры. Нет раздельных exit codes. Stack traces утекают к пользователю. i18n ключи для новых ошибок не добавлены. Нужна единая система: что сломалось / почему / что делать.

## Подход

1. Создать модуль форматирования ошибок с функцией formatUserError()
2. Добавить exit codes и --debug флаг
3. Добавить i18n ключи для всех новых ошибок на все 11 языков (en + ru обязательно, остальные fallback)

<task id="02-02-01">
<title>Создать модуль форматирования ошибок и exit codes</title>
<what>
Создать `src/core/errors.ts`:
- Функция `formatUserError(error: AdapterError | Error): string` — форматирует:
  ```
  ✘ Error: [что сломалось]
    Cause: [почему]
    Fix: [что делать]
  ```
  Для AdapterError использует поля agentName, code, suggestion.
  Для обычного Error — только message.
- Константы exit codes: EXIT_OK=0, EXIT_GENERAL=1, EXIT_SESSION_NOT_FOUND=2, EXIT_AGENT_NOT_INSTALLED=3, EXIT_CORRUPT_DATA=4
- Функция `getExitCode(error: AdapterError): number` — маппинг AdapterErrorCode → exit code
- Функция `handleFatalError(error: unknown, debug: boolean): never` — форматирует ошибку, показывает stack trace только если debug=true, вызывает process.exit с правильным кодом

Использовать i18n ключи (t('errorWhat'), t('errorCause'), t('errorFix')) для лейблов.
</what>
