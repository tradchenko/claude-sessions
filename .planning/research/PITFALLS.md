# Pitfalls: CLI Audit & Stabilization

*Контекст: brownfield Node.js CLI, ~42 TypeScript файла, 5 агент-адаптеров, npm-пакет у живых пользователей.*

---

## Refactoring Pitfalls

### Сломать работающее, пока чинишь другое

**Признаки раннего обнаружения:**
- Нет тестов — невозможно знать, что было рабочим до рефакторинга
- Изменение общего кода (утилиты, i18n, парсеры) затрагивает все 5 адаптеров сразу
- Смена типов/интерфейсов в `types.ts` ломает молчаливо (TypeScript ловит, но только при сборке)

**Стратегия:**
- Перед рефакторингом любого модуля — написать характеристические тесты (golden tests) на текущее поведение
- Один PR = один модуль, не смешивать рефакторинг нескольких адаптеров
- Монолитный `i18n.ts` (2132 строки) — трогать в последнюю очередь, только после покрытия тестами

**Фаза:** Phase 1 (аудит) — зафиксировать текущее поведение до любых изменений

---

### Scope creep — добавление фич вместо фиксов

**Признаки раннего обнаружения:**
- Появляются задачи вида "заодно добавим поддержку агента X"
- Рефакторинг адаптера превращается в полный редизайн API
- PR начинает затрагивать файлы за пределами аудируемого модуля

**Стратегия:**
- Жёсткий out-of-scope список зафиксирован в PROJECT.md — сверяться при каждом PR
- Если найдена хорошая идея — заводить issue с меткой `post-stabilization`, не брать в текущий milestone
- Код-ревью с проверкой: "это фикс бага или новая фича?"

**Фаза:** Постоянно, начиная с Phase 1

---

### Рефакторинг без тестов — нет страховочной сети

**Признаки раннего обнаружения:**
- Отсутствие `*.test.ts` файлов (текущее состояние проекта)
- "Я проверил руками" как единственная верификация
- Регрессии обнаруживаются пользователями, а не CI

**Стратегия:**
- Правило: нельзя менять модуль без написания теста на него первым
- Порядок: написать тест → убедиться, что он проходит → рефакторить → тест всё ещё проходит
- Начинать с наиболее рискованных модулей: парсеры JSONL, логика resume, postinstall

**Фаза:** Phase 1 — первые тесты до первого рефакторинга

---

### Big-bang перезапись vs инкрементальное улучшение

**Признаки раннего обнаружения:**
- Желание "переписать адаптеры с нуля, они все плохие"
- Ветка с рефакторингом живёт >2 недель без мержа
- Изменения охватывают >30% кодовой базы в одном PR

**Стратегия:**
- Strangler fig pattern: новый код живёт рядом со старым, постепенно замещает
- Максимум одного адаптера за итерацию
- Дублирующаяся логика адаптеров — вынести в общий модуль поэтапно, не за один раз

**Фаза:** Phase 2 (рефакторинг) — строгие границы PR

## Testing Pitfalls

### Тесты проходят, но тестируют не то

**Признаки раннего обнаружения:**
- Тест проверяет возврат `undefined` вместо конкретного значения
- `expect(result).toBeTruthy()` вместо `expect(result).toEqual({...})`
- Тест называется "should work" без описания конкретного случая

**Стратегия:**
- Тесты должны проверять конкретные выходные данные, не просто отсутствие исключений
- Code review тестов так же строго, как продуктового кода
- Мутационное тестирование для критических парсеров (хотя бы выборочно)

**Фаза:** Phase 1–2, установить стандарт с первых тестов

---

### Over-mocking — тестируем моки, не код

**Признаки раннего обнаружения:**
- Mock занимает больше строк, чем тестируемый код
- При изменении реализации тесты падают, хотя поведение то же самое
- `fs.readFile` мокируется там, где можно использовать tmp-директорию

**Стратегия:**
- Для файловых операций — использовать реальные tmp-директории (`os.tmpdir()`), не мокировать fs
- Мокировать только внешние агенты (вызовы Claude CLI, Codex CLI), не внутреннюю логику
- Разграничить unit (с моками) и integration тесты (с реальной ФС)

**Фаза:** Phase 1 — определить политику до написания тестов

---

### Пропущенные edge cases в JSONL/JSON парсинге

**Признаки раннего обнаружения:**
- Парсер падает на пустом файле или файле с одной строкой
- Не обрабатывается UTF-8 BOM в начале файла
- Последняя строка JSONL без `\n` не парсируется

**Конкретные edge cases для claude-sessions:**
- Повреждённая строка в середине JSONL (Claude, Codex, Qwen, Companion форматы)
- Сессия с нулевым числом сообщений
- JSONL файл, заблокированный агентом в момент чтения
- Файл `sessions.json` Claude с неполной записью (агент завершился аварийно)
- Gemini git repo без коммитов

**Стратегия:**
- Fixture-файлы с граничными случаями в `__fixtures__/` для каждого агента
- Все парсеры должны возвращать `Result<T, Error>`, не бросать исключения

**Фаза:** Phase 1 (аудит) — найти, Phase 2 — зафиксировать тестами и починить

---

### Platform-specific баги (macOS vs Linux пути)

**Признаки раннего обнаружения:**
- Хардкод `~/` вместо `os.homedir()`
- `path.join` vs конкатенация строк с `/`
- Предположение о регистре файловой системы (macOS case-insensitive)

**Конкретные риски проекта:**
- `~/.claude/settings.json`, `~/.codex/`, `~/.qwen/` — платформо-зависимые пути
- Windows WSL имеет другую структуру домашней директории
- Права доступа к файлам отличаются (chmod на Windows WSL)

**Стратегия:**
- Всегда `os.homedir()` + `path.join()`, никогда строковая конкатенация путей
- CI запускать на ubuntu-latest и macos-latest
- Отдельный тест: "все пути агентов резолвятся корректно на текущей платформе"

**Фаза:** Phase 1 — аудит путей, Phase 2 — рефакторинг

---

### Race conditions в файловых операциях

**Признаки раннего обнаружения:**
- Чтение и запись одного файла без блокировки
- `fs.existsSync` + `fs.readFileSync` как два отдельных вызова (TOCTOU)
- Параллельные операции с индексом памяти

**Стратегия:**
- Атомарные записи: писать во временный файл, затем `fs.rename()`
- Для MCP сервера памяти: очередь операций записи, не параллельные write
- Тесты с искусственными задержками для воспроизведения race conditions

**Фаза:** Phase 2–3, особенно при работе с memory store

## Migration Pitfalls

### Postinstall, ломающийся при частичном состоянии

**Признаки раннего обнаружения:**
- Postinstall не идемпотентен — второй запуск ломает первый результат
- Нет обработки кейса "установка поверх повреждённой предыдущей установки"
- Ошибка в postinstall = npm install завершается с ненулевым кодом = пользователь видит страшную ошибку

**Конкретные риски:**
- `~/.claude/settings.json` уже изменён пользователем вручную
- Хуки из предыдущей версии имеют другой формат (исторически были ломающие изменения)
- Частично записанный `settings.json` из прерванной предыдущей установки

**Стратегия:**
- Postinstall всегда завершается с кодом 0 — ошибки логируются, не пробрасываются
- Перед изменением файла — создать backup с timestamp
- Идемпотентность: повторный запуск даёт тот же результат

**Фаза:** Phase 3 (migration) — критичный приоритет

---

### Config migration, теряющая данные пользователя

**Признаки раннего обнаружения:**
- Миграция перезаписывает весь файл вместо патча конкретных полей
- Нет валидации после миграции
- Миграция запускается без резервной копии

**Стратегия:**
- Merge, не replace: читать существующий конфиг, патчить только управляемые поля
- Помечать управляемые секции комментарием/маркером в JSON
- Rollback: если после миграции валидация не прошла — восстановить из backup

**Фаза:** Phase 3

---

### Сбои определения версии

**Признаки раннего обнаружения:**
- `package.json` версия не соответствует установленной версии (кэш npm)
- Нет проверки версии существующих хуков перед миграцией
- Глобально установленный пакет имеет другую версию, чем локальная разработка

**Стратегия:**
- Записывать версию в managed-файлы при установке: `"_claude-sessions-version": "2.3.0"`
- Миграция принимает решение на основе этой версии, не на основе текущего `package.json`

**Фаза:** Phase 3

---

### Ограничения npm lifecycle hooks

**Признаки раннего обнаружения:**
- `postinstall` не запускается при `npm ci` в некоторых конфигурациях
- При глобальной установке (`npm i -g`) контекст выполнения отличается от локального
- `--ignore-scripts` флаг у пользователей полностью пропускает миграцию

**Стратегия:**
- Ленивая миграция: если postinstall не отработал — мигрировать при первом запуске команды
- Детектировать состояние при старте CLI, не только при установке
- Логировать предупреждение если обнаружена устаревшая конфигурация

**Фаза:** Phase 3 — проектировать с учётом этого ограничения

## Multi-Agent Pitfalls

### Предположение об одинаковом формате сессий у всех агентов

**Признаки раннего обнаружения:**
- Один парсер используется для нескольких агентов
- Поле `sessionId` берётся из одного места для всех форматов
- Тест написан только для Claude, остальные "такие же"

**Конкретные различия в проекте:**
- Claude: `sessions.json` (единый файл со всеми сессиями)
- Codex: отдельные JSONL файлы в `~/.codex/`
- Qwen: JSONL в `~/.qwen/`
- Gemini: git репозитории (принципиально другой формат)
- Companion: JSONL

**Стратегия:**
- Каждый адаптер — полностью изолированный модуль с собственными типами
- Общий `SessionAdapter` интерфейс, но нет общей реализации парсинга
- Матрица совместимости: тест на каждый агент × каждую операцию (list/resume/restore)

**Фаза:** Phase 1 — аудит, Phase 2 — рефакторинг адаптеров

---

### Хардкод путей, отличающихся по платформам

**Признаки раннего обнаружения:**
- `~/.claude` захардкожен как строка
- Путь к исполняемому файлу агента не проверяется через PATH
- Windows WSL пути не тестируются

**Стратегия:**
- Централизованный `AgentPaths` объект для каждого агента
- Пути — только через `os.homedir()` и `path.join()`
- Smoke test: "все пути существуют на тестовой машине"

**Фаза:** Phase 1 — обнаружить, Phase 2 — исправить

---

### Изменения CLI агентов, ломающие адаптеры

**Признаки раннего обнаружения:**
- Адаптер парсит вывод CLI команды, а не структурированный файл
- Нет версионирования ожидаемого формата
- Тест использует реальный агент (нет изоляции от внешних изменений)

**Стратегия:**
- Тестировать адаптеры против зафиксированных fixture-файлов, не против живых агентов
- Для resume/restore: читать файлы напрямую там, где возможно, вместо вызова CLI
- Документировать ожидаемый формат файлов для каждого агента с версией

**Фаза:** Phase 1 — аудит зависимостей от CLI агентов

---

### Тестирование против реальных агентов vs моки

**Признаки раннего обнаружения:**
- CI падает потому что Claude Code не установлен на CI-машине
- Тест требует активной сессии агента
- Разные версии агента дают разный вывод

**Стратегия:**
- Unit тесты: только fixtures, ноль реальных агентов
- Integration тесты: отдельный набор, запускается вручную или в специальной среде
- Fixture файлы коммитятся в репозиторий и версионируются

**Фаза:** Phase 1 — установить правило до написания тестов

## Memory System Pitfalls

### LLM extraction даёт непоследовательные результаты

**Признаки раннего обнаружения:**
- Одна и та же сессия при повторном извлечении даёт разные воспоминания
- Формат выходных данных LLM меняется между версиями модели
- Извлечение зависит от температуры модели (недетерминировано)

**Конкретный риск:**
- L1 (LLM semantic) extraction в claude-sessions нестабильна по определению
- Смена модели Claude может изменить формат всех новых воспоминаний

**Стратегия:**
- Строгая JSON-схема для выхода LLM с валидацией
- Если валидация не прошла — fallback к L0 (quick metadata), не ошибка
- Тесты для extraction используют детерминированные mock-ответы LLM

**Фаза:** Phase 1 — аудит схемы валидации

---

### Hotness decay делает всё холодным

**Признаки раннего обнаружения:**
- После нескольких дней все воспоминания имеют hotness = 0
- Decay применяется при каждом чтении, а не по расписанию
- Нет минимального порога hotness для retention

**Стратегия:**
- Decay только по времени (cron-like при запуске), не при каждом доступе
- Minimum floor: hotness не падает ниже порога для "core" memories
- Мониторинг: логировать распределение hotness при старте

**Фаза:** Phase 1 — аудит алгоритма, Phase 2 — исправление

---

### Дедупликация слишком агрессивная или слишком мягкая

**Признаки раннего обнаружения:**
- Дублирующиеся воспоминания накапливаются быстрее чем ожидается
- Уникальные воспоминания объединяются и теряют контекст
- Дедупликация основана на точном совпадении текста (не семантическом)

**Стратегия:**
- Гибридный подход: exact match для идентичного текста + threshold для семантически близких
- Сохранять оба воспоминания с пометкой "similar to X" вместо слияния
- Тест: набор fixture-воспоминаний с ожидаемым результатом дедупликации

**Фаза:** Phase 1 — аудит логики, Phase 2 — тесты и исправление

---

### Повреждение индекса от конкурентного доступа

**Признаки раннего обнаружения:**
- MCP сервер и CLI команда обращаются к индексу одновременно
- Нет file locking при записи в индекс
- После аварийного завершения индекс в несогласованном состоянии

**Стратегия:**
- Atomic writes: write-to-temp + rename для всех операций записи в индекс
- Advisory file lock (`.lock` файл) перед любой модификацией
- Startup integrity check: если индекс повреждён — пересобрать из источников

**Фаза:** Phase 2–3, при рефакторинге memory store
